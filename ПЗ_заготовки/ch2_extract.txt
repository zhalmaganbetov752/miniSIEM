168|2 Выбор технологий и средств разработки
169|Выбор технологий для дипломного mini SIEM определяется одновременно учебными целями (прозрачность кода, воспроизводимость стенда на одном ноутбуке), ограничениями доступа к корпоративной инфраструктуре оператора на время практики и инженерными компромиссами между простотой эксплуатации и достаточной выразительностью средств анализа событий.
170|Язык программирования и среда исполнения. В качестве основного языка выбран Python версии 3.11 (разрядность x64), рекомендованная в документации проекта для совместимости с используемыми колёсами и PyInstaller. Python обеспечивает высокую скорость разработки прототипов, богатую стандартную библиотеку для работы с текстом, временем, структурами данных и многопоточностью, а также зрелую интеграцию с Windows через пакет pywin32 для доступа к Win32 API журналов событий. Для задач сбора логов доминирует режим ожидания ввода-вывода; интерпретируемый язык при этом не становится узким местом при пакетном чтении и аккуратной записи в SQLite.
171|Архитектурный стиль (почему именно так). Для mini SIEM выбран модульный стиль с разделением «сбор → хранение → анализ → представление». В дипломной работе это важно по двум причинам. Во‑первых, модульность позволяет показать понимание промышленной архитектуры SIEM, где сборщики и хранилища часто независимы и масштабируются по-разному. Во‑вторых, она упрощает сопровождение и демонстрацию: при ошибке в UI не страдает конвейер хранения, а при временной недоступности источника события не разрушается логика правил и отчётности. В проекте такая декомпозиция выражается в наличии отдельных модулей для базы данных (`db.py`), правил (`rules.py` и YAML-файл), чтения журналов Windows (`collector_win.py`), импорта архивов (`evtx_import.py`), отчётности (`report.py`) и графического приложения (`app.py`).
172|Обоснование выбора локального контура. В операторской среде большинство критичных логов и сетевых событий находится в зоне ограниченного доступа (коммерческая тайна, персональные данные, требования регулятора). Поэтому для дипломного проекта целесообразно использовать локальный контур, который можно легально воспроизвести на стенде без доступа к production. Журналы Windows позволяют отработать ключевые классы событий ИБ (аутентификация, запуск процессов, системные изменения), а также показать, что «SIEM-логика» универсальна и переносима на сетевые источники при дальнейшем развитии.
173|Доступ к журналам Windows. Классические журналы Security, System и Application читаются через устоявшийся Win32-механизм; для канала вида `Microsoft-Windows-Sysmon/Operational` требуется API подсистемы Windows Event Log (Evt), поскольку «длинные» каналы не обслуживаются legacy-функциями OpenEventLog так же универсально. Разделение путей чтения в одном модуле снижает риск ошибок конфигурации и упрощает сопровождение. Нормализация идентификатора события с маской младших 16 бит важна для корректной работы правил, опирающихся на «канонические» номера событий Microsoft.
174|Сбор, нормализация и качество данных. Качество детектирования зависит от того, насколько стабильно и одинаково интерпретируются поля событий при разных версиях Windows и при разных источниках (классический журнал против Sysmon). Для этого в mini SIEM применён подход «raw-normalized»: события приводятся к минимальному общему набору полей (время, канал, provider, EventID, level, computer, record_id, message) и дополняются JSON-полем с исходными деталями. Такой подход сохраняет доказательную базу (можно вернуться к исходным вставкам), при этом делает возможными универсальные SQL-запросы для фильтрации и правил.
175|Почему важны разобранные поля для 4624/4625. Для событий входа (Security 4624/4625) в реальной аналитике значимы атрибуты: учётная запись, тип входа (в т.ч. удалённые сценарии), IP-адрес источника. В Windows часть этих данных находится в массиве строковых вставок, формат которых зависит от локали и версии. В проекте предусмотрено хранение как сырого массива вставок (через JSON), так и отдельных разобранных полей (parsed_account, parsed_ip, parsed_logon_type) для ускорения запросов и корректного группирования по IP/аккаунту. Это уменьшает «стоимость» корреляционных запросов и делает правила менее зависимыми от формата сообщения.
176|СУБД и режим хранения. В качестве встраиваемого хранилища выбран SQLite. Причины: отсутствие необходимости поднимать отдельный сервер СУБД на учебном стенде; переносимость одного файла базы; достаточная для диплома выразительность SQL для агрегаций и оконных выборок; поддержка индексов и внешних ключей для целостности связи алертов и действий. Режим журнала WAL снижает вероятность блокировок при одновременной записи событий сборщиком и чтении таблиц интерфейсом. Параметры PRAGMA synchronous = NORMAL и включённые foreign_keys соответствуют разумному балансу между производительностью и устойчивостью на рабочей станции. Размещение файла базы в каталоге данных пользователя (`LOCALAPPDATA\miniSIEM` в типовой конфигурации) соответствует модели разграничения: данные привязаны к учётной записи ОС и не требуют прав записи в защищённые системные каталоги.
177|Схема данных и практическая мотивация таблиц. Наличие в БД отдельных таблиц `events`, `alerts` и `actions` отражает практику эксплуатации (и наблюдения на практике в ИТ-отделе): важно не только «увидеть алерт», но и зафиксировать, что с ним делали. Таблица `alerts` позволяет реализовать статусы обработки (новый, в работе, закрыт) и комментарии, а `actions` — сохранять след реакции (пусть даже в dry-run): когда действие предложено/выполнено, к какому алерту относится, какой результат и детали. Для диплома это важно как демонстрация процесса «обнаружение → разбор → решение», а не только «детектирование ради детекта».
178|Политика хранения и контроль размера БД. В SIEM-системах всегда присутствует вопрос ретенции: сколько хранить в «горячем» доступе и как очищать старые данные. В локальном SQLite это выражается в необходимости предусмотреть операции обслуживания: удаление данных старше заданного срока, checkpoint WAL и vacuum. Эти операции важны, чтобы файл базы не рос бесконтрольно, а также чтобы интерфейс оставался отзывчивым при длительной работе. В отчёте уместно указать, что политика хранения является параметром стенда и должна согласовываться с требованиями к хранению событий (для учебного стенда — по возможности не менее нескольких дней для демонстрации динамики и трендов).
179|Сравнение с альтернативами хранения. Elasticsearch или OpenSearch дали бы полнотекстовый поиск и горизонтальное масштабирование, но потребовали бы JVM/кластера, политик индексов и отдельной эксплуатации — что выходит за рамки целевой простоты дипломного стенда. PostgreSQL с партиционированием по времени являлся бы сильным компромиссом для «одного сервера без кластера», но для локального single-user приложения SQLite остаётся более прямолинейным выбором. При эволюции проекта слой доступа к данным можно абстрагировать и заменить бэкенд, сохранив модель события и правил.
180|Критерии выбора «простота против масштабирования». Дипломный проект намеренно выбирает минимально достаточную инфраструктуру. Это снижает трудозатраты и риски на этапах демонстрации (не требуется отдельный сервер, сеть, права администрирования кластера). При этом в тексте отчёта важно показать понимание того, как проект масштабировался бы при изменении требований: вынесение хранилища на сервер, централизованный коллектор, брокер сообщений для буферизации пиков, поддержка нескольких агентов и центральной консоли. Такой «планируемый путь развития» является хорошей практикой проектирования.
181|Описание правил и конфигурация. Формат YAML выбран для человекочитаемости правил аналитиком и для удобного версионирования в системе контроля версий. Правило содержит стабильный идентификатор, заголовок, описание, уровень серьёзности, имя детектора и блок параметров. Такой подход разделяет код движка детектирования и политику организации (какие сценарии включены и с какими порогами), что соответствует лучшим практикам управляемости SIEM.
182|Почему правила параметризуются. В практической эксплуатации пороги «сколько ошибок входа считать подозрительными» не являются универсальными: офис с редкими входами и терминальный сервер с множеством сервисных аккаунтов имеют разные базовые уровни событий. Поэтому в YAML вынесены параметры `threshold` и `window_minutes`. Это позволяет демонстрировать тюнинг правил на защите: сначала показать чувствительную конфигурацию, затем объяснить риск ложных срабатываний и продемонстрировать корректировку порога.
183|Детерминированность и воспроизводимость срабатываний. Для дипломной работы важно, чтобы срабатывания правил можно было воспроизвести в аудитории. Поэтому предусмотрены либо синтетические сценарии (например, генерация нескольких неудачных входов на тестовой ВМ), либо импорт EVTX с заранее подготовленным набором событий. На уровне технологий это оправдывает наличие `evtx_import.py` и отчётных форматов, которые можно приложить к пояснительной записке без доступа к исходной машине.
184|Пользовательский интерфейс. Для десктопной Windows-среды выбран фреймворк Qt через официальные биндинги PySide6: нативный внешний вид, табличные представления для больших объёмов событий, встроенная поддержка печати и диалогов, зрелая модель сигналов и слотов для взаимодействия с фоновыми потоками. Альтернатива в виде веб-интерфейса (FastAPI + React) увеличила бы объём фронтенд-разработки и усложнила бы демонстрацию на защите без поднятого сервера; для целей диплома десктопное приложение является прагматичным решением.
185|Требования к UX аналитика. Даже в mini SIEM пользовательский интерфейс должен решать типовые задачи аналитика: быстро отфильтровать события по времени и каналу, найти конкретный EventID, посмотреть контекст сообщения, перейти от алерта к связанным событиям, отметить статус обработки и добавить комментарий. Поэтому технологии UI выбирались не «ради красивого окна», а ради производительных таблиц, сортировок и механизмов фоновой загрузки, чтобы не блокировать окно при обновлении данных.
186|Упаковка и поставка. PyInstaller позволяет собрать один исполняемый файл miniSIEM.exe, что упрощает перенос на машину без установленного Python и демонстрацию работы комиссии. Ограничением является необходимость тестирования сборки на целевой ОС и учёт ложных срабатываний антивируса на неподписанный exe; в пояснительной записке целесообразно упомянуть возможность цифровой подписи как этап промышленной зрелости.
187|Почему важна поставка в виде одного файла. Для защиты диплома и демонстрации преподавателю/рецензенту критично минимизировать «факторы среды»: несовпадение версий Python, отсутствие нужных библиотек, запреты корпоративной политики на установку софта. Standalone exe уменьшает эти риски и делает демонстрацию воспроизводимой. При этом в отчёте корректно указать, что промышленная поставка обычно требует подписи, политики обновлений и контроля доверия, а в дипломе реализуется упрощённый вариант.
188|Библиотеки и инструменты разработки. Использование pywin32 для событий Windows; PyYAML для разбора правил; стандартный sqlite3; PySide6 для GUI; PyInstaller для релизной сборки. Среда разработки — виртуальное окружение Python, файл requirements.txt, система контроля версий Git. Тестирование — ручное на стенде с генерацией трафика входа и импортом EVTX.
189|Контроль качества кода и проверяемость. В дипломной работе важно показать, что проект сопровождаем и пригоден к развитию. Для этого применяются: единая структура пакета, изоляция зависимостей в venv, версионирование, а также минимальные проверки (запуск на «чистой» машине, повторная установка зависимостей, тест импортов EVTX). Даже если автотесты покрывают не весь функционал, описанная методика проверки повышает доверие к результату.
190|Производительность и многопоточность. Сбор событий и тяжёлые запросы к БД не должны блокировать главный поток UI; в проекте используется вынесение операций в отдельные потоки Qt с сигналами завершения. Пакетный размер чтения из журнала ограничивается, чтобы сгладить пики при всплеске активности ОС.
191|Методика оценки производительности (для текста отчёта). В качестве простого критерия можно использовать: скорость импорта EVTX (количество событий в минуту), среднее время обновления таблицы событий за окно в 24 часа, время формирования отчёта, а также рост файла базы при типовой нагрузке. Эти метрики легко воспроизводимы и подходят для описания результатов дипломной работы без доступа к инфраструктуре предприятия. Для правил burst важна амортизированная стоимость SQL-запроса в скользящем окне; здесь роль играют индексы по времени и каналам.
192|Безопасность разработки и эксплуатации. Чтение журнала Security требует привилегий; проект должен документировать ожидаемую модель прав. Реакции на алерты по умолчанию выполняются в режиме симуляции; любые реальные действия требуют явного включения и понимания риска. Файл SQLite содержит потенциально чувствительные данные (учётные имена, IP); доступ к нему должен контролироваться политиками ОС и шифрованием диска на уровне рабочей станции.
193|Конфиденциальность и комплаенс в учебном стенде. При работе со событиями Windows возможна фиксация персональных данных (имена пользователей домена, IP-адреса). Поэтому в отчёте уместно указать, что демонстрации проводятся на тестовом стенде, а экспорт отчётов осуществляется без данных реальных клиентов. Это соответствует принципам минимизации данных и корректного обращения с журналами.
194|Управление доступом к артефактам. Даже при локальном хранении следует учитывать: файл SQLite и отчёты HTML/JSON являются артефактами расследования. В промышленной практике они должны храниться в защищённом каталоге, а доступ — ограничиваться. В дипломной работе достаточно обозначить этот принцип и рекомендовать хранение на диске с шифрованием и раздельными учётными записями ОС.
195|Связь выбора технологий с наблюдениями практики. В ИТ-отделе оператора подчёркивается значение единой шкалы времени, идентификации источника и неизменяемости журналов насколько это возможно политиками. Локальный SQLite-файл с UTC-метками и таблицей действий по алертам воспроизводит эту логику в учебном масштабе. Принцип «сначала зафиксировать, потом анализировать» совпадает с культурой заполнения тикетов при разборе обращений клиентов на практике.

221|Среда разработки и воспроизводимость. Рекомендуется фиксировать версии зависимостей в `requirements.txt` и использовать виртуальное окружение, чтобы исключить дрейф библиотек между машинами студента и научного руководителя. Для отчёта полезно указать используемую ОС сборки (например, Windows 10 или Windows 11) и разрядность, поскольку pywin32 и PyInstaller чувствительны к платформе.
222|Кодировки и локализация. Сообщения журналов Windows могут быть локализованы; детекторы, опирающиеся на подстроки в командной строке, должны учитывать риск ложных срабатываний при другом языке интерфейса. В обосновании технологий фиксируется, что для учебного стенда допустима настройка тестовой системы на одной локали; для промышленного развития потребовались бы нормализованные поля или multilingual-паттерны.
223|Модель угроз на уровне ОС и обоснование Sysmon. Встроенный аудит Windows даёт базовую картину входов и (при включении) создания процессов, но Sysmon расширяет детализацию (хэши, родительский процесс, командная строка, сетевые соединения — в зависимости от конфигурации XML). В разделе выбора технологий Sysmon рассматривается как опциональный, но рекомендуемый источник для детектов запуска PowerShell с EncodedCommand и аналогичных сценариев. Установка Sysmon требует административных прав и отдельной политики организации; в дипломной документации это должно быть явно оговорено.
224|Индексация и запросы. Выбор индексов в SQLite (по времени, по паре канал-время, по паре event_id-время) напрямую следует из профиля запросов детекторов: большинство правил работает в скользящем окне по времени и фильтрует по каналу Security или по конкретным EventID. Обоснование такого набора индексов — снижение амортизированной стоимости каждого цикла опроса правил при росте таблицы событий.
225|Графический интерфейс и отчётность. Qt через PySide6 даёт табличные модели для больших выборок, фильтры, печать и экспорт. Модуль отчётности в HTML/JSON позволяет приложить к отчёту по практике или к пояснительной записке артефакты разбора без скриншотов базы вручную. С точки зрения ИБ экспорт должен рассматриваться как чувствительный артефакт и передаваться по защищённым каналам.
226|Альтернатива: веб-интерфейс. Если бы целью было одновременное наблюдение несколькими аналитиками, логичнее был бы веб-клиент и сервер приложений. Отказ от этой модели в пользу десктопа снижает трудоёмкость и поверхность атаки (нет открытого HTTP-порта по умолчанию), что для учебного single-user приложения является разумным компромиссом.
227|Лицензирование компонентов. PySide6 распространяется под LGPL; SQLite находится в общественном достоянии; Python — PSF license. Для дипломной работы важно отметить юридическую совместимость стека и отсутствие необходимости закупки коммерческих лицензий SIEM для демонстрации компетенций.
228|Пути развития стека после диплома. Экспорт нормализованных событий в syslog или JSON Lines на локальный коллектор; замена SQLite на PostgreSQL при необходимости многопользовательского доступа; добавление брокера сообщений для буферизации пиков; подключение внешних списков IOC; интеграция с Telegram или почтой для доставки алертов. Эти пункты не входят в минимальные критерии приёмки, но показывают понимание эволюции архитектуры.
229|Связь с практикой в операторе (повторение на уровне технологий). Наблюдаемая на практике зависимость качества разбора инцидента от полноты записей в тикете напрямую мотивирует проектирование таблицы `actions` и комментариев к алертам в mini SIEM: инструмент аналитики должен поддерживать след расследования, а не только момент срабатывания правила.
